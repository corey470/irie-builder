Grading rubric for "Currently in Tuner?":

- **yes** — an affordance in the current Tuner lets the user reach the same user-visible outcome (even if UX differs).
- **partial** — something related exists but behavior is reduced (e.g. the concept is exposed as a dial, but the direct per-object input is gone; or the control exists but writes to a different store).
- **no** — the capability is stripped; no Tuner equivalent.

All file:line references are pre-Tuner positions in `app/_components/builder-platform.tsx` at `5241dc5`. Pre-Tuner here means "latest state of the monolith `/edit` surface before the Tuner cutover" — since the monolith wasn't touched by the Tuner work, these lines are still present in the file but no longer reachable via the routed `/edit` page.

## Top user-impact "no" capabilities (restoration priority, subjective)

1. **Inline contentEditable text editing** — without it, users can't change copy in `/edit` at all. This is the #1 reason the page feels broken. (`builder-platform.tsx:2947–2956`, `3141–3149`)
2. **Click-preview-to-select text/image** — the primary navigation affordance; currently only sections respond to clicks. (`builder-platform.tsx:3113–3138`)
3. **Replace-image file picker** — no way to swap images at all today. (`builder-platform.tsx:3009–3049`)
4. **Accent color picker (top-bar popover)** — brand alignment is a single click in the old editor and impossible in the new one. (`builder-platform.tsx:3336–3341`)
5. **Text Size / Line Height / Letter Spacing sliders** — typography tuning is the most-used object-style set; missing entirely. (`builder-platform.tsx:3522–3553`)
6. **Text color picker + Align segmented** — basic text styling, not addressable by any Tuner dial. (`builder-platform.tsx:3611–3644`)
7. **Section Background Color / Background Image / Overlay Opacity** — hero-section branding knobs; Tuner's accent-alpha dial is not a substitute. (`builder-platform.tsx:3445–3464`)
8. **Fullscreen preview button** — was in the old top-area; standard expectation for a site editor. (`builder-platform.tsx:3355–3358`, `1376–1401`)
9. **CTA button Radius / Padding / Background / Border sliders** — per-CTA overrides, not section-level. (`builder-platform.tsx:3572–3693`)
10. **Image Fit / Position / Radius controls** — image behavior knobs; without these a replaced image often looks wrong. (`builder-platform.tsx:3719–3750`)

## Notes / ambiguities

- **`/publish` vs export dropdown.** Pre-Tuner the Editor page had both a "Download HTML" button and an `Export →` link to `/publish`. Tuner's top-bar has an Export dropdown with "Download HTML", "Publish", and "Copy code"; the "Publish" item currently calls `handleDownload` (`TunerTopBar.tsx:335–343`, wired from `TunerEditor.tsx:729–731`). Flagging — the Publish route may still exist and may need to be reconnected. Not in scope for Phase 2 unless Corey wants the route back.
- **Pre-Tuner Download HTML called `logPublish`; Tuner's `handleDownload` does not.** Consider re-adding during object-mode restore so `builder_publishes` rows resume.
- **Absolute px padding vs scale multiplier.** The old Section inspector exposed absolute px padding sliders; the Tuner exposes a `padding` dial as a scale multiplier (0.5×–2× on `--irie-pad-scale`). Not strictly equivalent — scale multiplies the section's designed padding; absolute px overrides it. Per the brief this row is treated as **partial** and marked for restoration of the absolute sliders under Section-style mode. If Corey wants the scale dial to subsume this, mark the restore row complete and drop the two absolute sliders.
- **Font family / weight selects.** Pre-Tuner shipped these as plain selects. Could be reframed as Tuner dials (chips) for design consistency, but the immediate restore per brief is a plain select — flagged as an ambiguity only because it's a design decision, not an engineering one.
- **`FONT_FAMILY_OPTIONS`** (pre-Tuner) lists only three options: Playfair Display, Syne, System Sans (`builder-platform.tsx:746–750`). Small set — worth confirming whether a future design expansion is planned before restoring 1:1.
- **Box-shadow free-text field.** The CTA "Shadow" control is a free-text CSS input — the only typed-CSS affordance in the pre-Tuner inspector. Restoring it as-is will feel out of place next to sliders. Considered below "Advanced" disclosure but flagging for Corey's call.
- **Mobile bottom dock: Theme sheet.** Pre-Tuner had a Theme sheet on mobile; no theme concept exists in Tuner today. If the accent popover moves to the top bar, a dedicated mobile Theme sheet may be unnecessary — roll the accent into the single `TunerMobileDrawer` instead. Flagging.
- **Undo/redo across object + dial edits.** The Tuner history stack currently only tracks `tunerState`. Object-mode text/image/accent/style edits will need to either (a) live in the same snapshot so undo traverses both, or (b) maintain a second stack — choice affects UX coherence. Flagging as a Phase-2 design decision.
- **Storage-backed image uploads.** Pre-Tuner stored replaced images as base64 `data:` URLs inside `final_html`. This is the audited pre-Tuner behavior and the row should be restored as-is. Lifting to Supabase Storage is a future task, not a restore.
- **Iframe document `doc.open()/doc.write()` on accent change.** Pre-Tuner accent picker rewrites the entire iframe srcDoc on every change (`builder-platform.tsx:2845–2852`). Tuner's runtime-CSS approach is gentler (mutates CSS vars without reload). Restoring 1:1 will cause a reflow flash on every accent change and will lose Tuner's section rings briefly. Recommend re-installing runtime CSS + re-annotating sections after `doc.write`, or replacing the rewrite with a `replaceColorTokens`-free CSS-var approach if the generator can be made to use `var(--irie-accent)`. Flagging as a design concern.
